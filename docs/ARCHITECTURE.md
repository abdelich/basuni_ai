# Архитектура ботов BasuniAI (Discord)

## Структура проекта (оркестратор и роли)

```
basuniAI/
  main.py                 # Точка входа: python main.py
  config/
    default.yaml          # guild_id, channels, role_ids, enabled_roles, roles.*
  prompts/                # Системные промпты по одному файлу на роль
    elder_system.md
    # council_system.md, prosecutor_system.md, ...
  src/
    core/                 # Общее ядро
      config.py           # Загрузка .env + YAML
      db.py               # Async SQLAlchemy, get_db(), init_db()
      models.py           # Общие модели (cases и др.)
    orchestrator/
      runner.py           # run(): загрузка конфига, init_db, запуск всех enabled_roles
    roles/
      base.py             # RoleBot, RoleDeps, RoleDescriptor, register
      __init__.py         # ROLE_REGISTRY, load_all_roles()
      elder/
        __init__.py       # Регистрация роли "elder"
        bot.py            # ElderBot, create_elder_bot()
        logic.py          # Типы дел, допустимые решения, проверки полномочий
      # council/, prosecutor/, kpp/, judge/ — по той же схеме
  data/                   # SQLite (если не задан DATABASE_URL)
  .env                    # Секреты (по образцу .env.example)
```

- **Одна роль = один модуль** в `src/roles/<role_key>/`: свой бот, своя логика, свой промпт в `prompts/<role_key>_system.md`.
- **Оркестратор** не знает конкретных ролей: читает `enabled_roles` из конфига и из реестра `ROLE_REGISTRY` получает фабрику бота. Добавление новой роли = новый модуль в `roles/` + регистрация в `load_all_roles()` + запись в `enabled_roles` и токен в `.env`.

---

## Агент, инструменты, каналы

- **Один бот = один агент**: всё проходит через языковую модель (LLM). Пользователь пишет в канал → бот передаёт сообщение в LLM с системным промптом роли и списком **инструментов** (function calling) → модель решает, что ответить и какие инструменты вызвать → бот выполняет вызовы (Discord, БД) и при необходимости повторяет запрос к LLM → итоговый ответ отправляется в Discord.

- **Инструменты у каждой роли свои**: старейшина может публиковать решение, уведомлять суд/совет, читать дело из БД; совет — исполнять решения, мутить и т.д. Инструмент = имя + описание + параметры (JSON Schema) + асинхронная функция `execute(**kwargs) -> str`. Результат возвращается в LLM как контекст.

- **Привязка каналов** задаётся в `config/default.yaml`:
  - `channels`: словарь «ключ канала → ID» (например `elder_inbox`, `elder_decisions`, `court_inbox`).
  - В блоке `roles.<role>` задаются назначения: `inbox_channel_key`, `decisions_channel_key`, `outbox_channel_key`, `notify_court_channel_key`, `notify_council_channel_key`. Так бот знает, **откуда** брать заявки (inbox), **куда** публиковать решения (decisions), **куда** отвечать гражданину (outbox, часто тот же inbox), **куда** слать уведомления суду/совету.

- Граждане пишут в канал-inbox роли; бот обрабатывает сообщение агентом и отвечает в тот же канал (или в тред). Публикация решений — в канал decisions через инструмент `publish_decision` и т.д.

- **Категория «право» и прецеденты**: в конфиге задаётся `reference_category_name` (по умолчанию «право»). Подканалы этой категории содержат прецеденты и закон. **Для каждого агента** должна быть одна и та же логика: читать эти каналы (инструмент `get_channel_content(channel_id)`) и при необходимости ссылаться на закон в ответах. В системном промпте каждой роли указывается, что категория «право» и её подканалы нужно учитывать и использовать для обоснования.

---

## Общая идея

- **Один оркестратор** — приложение, которое поднимает и управляет всеми ботами.
- **Один бот = одна роль по конституции** — у каждого бота своя роль (старейшина, совет, прокурор, КПП, судья), своя база полномочий и свой промпт для GPT.
- **Единое хранилище дел** — дела, апелляции, референдумы, прецеденты хранятся в одной БД; боты только читают/пишут в разрешённые им сущности.
- **Проверка полномочий** — перед действием бот проверяет: 1) роль пользователя в Discord; 2) тип дела/запроса; 3) что действие разрешено конституцией для этой роли.

Дальше в документе — детализация **только для бота «Старейшина»**.

# Бот «Старейшина» (Elder)

## Источники в конституции

- **Статья IV** — Старейшины (основные полномочия и ограничения).
- **Статья XV** — Референдум (роль старейшин при равных голосах).

---

## 1. Полномочия старейшин (что бот МОЖЕТ)

### 1.1. Рассматривать только три типа дел

| Тип дела | Описание | Источник |
|----------|----------|----------|
| **Апелляция по процедуре** | Жалоба на нарушение процедуры суда (сроки, кворум, конфликт интересов и т.д.) | Ст. IV, п. 2 |
| **Дело о референдуме** | Запрос гражданина с ПМЖ или прокурора — проводить ли референдум по вопросу | Ст. II, п. 12; Ст. IV, п. 2; Ст. XV, п. 2 |
| **Дело, не установленное судом** | Суд не смог принять решение (разногласие судей) — старейшины задают фактическую рамку и возвращают в суд | Ст. II, п. 6; Ст. IV, п. 2, 7 |

### 1.2. При рассмотрении апелляции — проверять

- соблюдение процедуры суда;
- наличие кворума присяжных (не менее 66 %);
- отсутствие конфликта интересов;
- соблюдение срока рассмотрения дела (максимум 16 часов).

### 1.3. По результатам апелляции — три возможных действия

1. **Подтвердить корректность** судебного процесса (апелляция отклонена).
2. **Передать решение на исполнение в Совет** (если решение старейшин требует исполнения — например, «процесс корректен, совет исполняет решение суда»).
3. **Однократно вернуть дело в суд** с указанием процедурного нарушения (повторно по тому же делу старейшины вмешиваться не вправе).

### 1.4. По делу «не установлено судом»

- установить фактическую рамку дела;
- обеспечить возможность вынесения судом решения (передать дело обратно в суд с уточнёнными рамками).

### 1.5. По делам о референдуме

- решить: проводить референдум по запросу гражданина/прокурора или нет;
- при равных голосах в уже проведённом референдуме — внести изменения в формулировку и передать суду на повторное проведение (суд обязан провести).

---

## 2. Ограничения (чего бот НЕ МОЖЕТ)

- Отменять судебные решения.
- Отменять судебные прецеденты.
- Пересматривать право по существу дела.
- Повторно вмешиваться в одно и то же дело (только один возврат в суд по апелляции).
- Выносить приговоры и формировать прецеденты.
- Рассматривать что-либо, кроме: апелляции по процедуре, дела о референдуме, дела не установленного судом.

---

## 3. Действия бота на сервере Discord

Технически бот может (в рамках своей роли):

| Действие | Назначение |
|----------|------------|
| Читать каналы | Канал обращений к старейшинам, канал дел/апелляций, канал референдумов (если есть). |
| Создавать/редактировать сообщения | Публичные постановления, отказы, запросы документов, уведомления о передаче в суд/совет. |
| Создавать треды | Одно дело = один тред (опционально) для истории и привязки к делу. |
| Реагировать (эмодзи) | Подтверждение получения обращения, статус «принято к рассмотрению», «передано в суд/совет». |
| Упоминать роли/пользователей | Вызов суда, совета, прокурора, уведомление заявителя. |
| Не применять муты/баны/роли | Наказания и выдача ПМЖ — не в полномочиях старейшин; только решения и передачи. |

То есть бот «Старейшина» **не** выдаёт роли, не мутит, не банит — только приём обращений, рассмотрение, текстовые решения и уведомления/передача дел в суд или совет.

## 4. Команды для уполномоченных (кто может вызывать)

По конституции к старейшинам обращаются:

- граждане с ПМЖ (апелляция по процедуре, запрос референдума);
- суд — передаёт дело «не установленное судом»;
- прокурор — передаёт вопрос о референдуме.

Предлагаемый набор команд (доступ только при наличии роли **ПМЖ** или **старейшина** для служебных действий):

| Команда | Кто может | Описание |
|---------|-----------|----------|
| `/appeal [номер дела] [причина]` | ПМЖ | Подать апелляцию по процедуре по указанному делу. |
| `/referendum_request [вопрос] [краткое обоснование]` | ПМЖ | Запросить проведение референдума (запрос уходит старейшинам). |
| `/elders_status [номер дела]` | ПМЖ, старейшина | Показать статус дела у старейшин (принято / в работе / решение вынесено). |
| `/elders_list` | ПМЖ, старейшина | Список дел, находящихся у старейшин (отфильтровано по типу). |

Служебные (только для старейшин, опционально через отдельный бот или те же команды с проверкой роли):

- Принять дело к рассмотрению.
- Вынести решение: подтвердить процесс / передать в совет / вернуть в суд с нарушением.
- По референдуму: одобрить/отклонить проведение; при равных голосах — отправить на повтор с изменённой формулировкой.

Права на команды проверяются по роли в Discord (ПМЖ, старейшина).

## 5. GPT: роль и использование

- **Роль**: «Ты — старейшина. Ты рассматриваешь только апелляции по процедуре, дела о референдуме и дела, не установленные судом. Ты не отменяешь решения и прецеденты, не выносишь приговоров. Твои решения: подтвердить процесс, передать на исполнение в совет или один раз вернуть дело в суд с указанием процедурного нарушения.»
- **Вход**: тип дела, текст заявления, данные из БД (суд, присяжные, сроки, кворум, конфликт интересов — что есть).
- **Выход**: структурированное решение (тип: confirm / send_to_council / return_to_court + текст обоснования). Финальное действие на сервере (пост, передача дела) выполняет бот по этому решению, а не «от имени» GPT (безопасность и предсказуемость).

## 6. Как это реализовать (кратко)

### 6.1. Стек

- **Язык**: Python.
- **Discord**: `discord.py` (или `nextcord`).
- **БД**: SQLite/PostgreSQL — дела, апелляции, референдумы, решения старейшин, связь с Discord (id каналов, сообщений, пользователей).
- **GPT**: OpenAI API (или аналог) — один сервис с разными системными промптами для каждого бота.

### 6.2. Общая схема

1. **Оркестратор** запускает N инстанций ботов (старейшина, суд, совет, прокурор, КПП); у каждого свой токен и своя «роль» в коде.
2. **Единый слой дел** (таблицы): дела суда, апелляции, референдумы, решения старейшин. Бот «Старейшина» читает/пишет только в разрешённые таблицы (апелляции, решения старейшин, референдумы — в части, относящейся к старейшинам).
3. **Бот «Старейшина»**:
   - слушает команды и сообщения в «приёмном» канале;
   - проверяет роль (ПМЖ / старейшина);
   - создаёт или привязывает запись в БД;
   - при необходимости запрашивает у GPT структурированное решение по делу;
   - публикует решение в Discord и обновляет статус дела, при необходимости шлёт уведомление в канал суда/совета (упоминание или сообщение в нужный канал).
4. **Запрет двойного вмешательства**: в БД у каждого дела — флаг «старейшины уже выносили решение по апелляции»; при попытке повторного рассмотрения бот отказывает.

### 6.3. Что сделать первым для старейшины

1. Описать схему БД: дела, апелляции, решения старейшин, референдумы.
2. Реализовать бота с командами `/appeal`, `/referendum_request`, `/elders_status`, `/elders_list` и проверкой роли ПМЖ.
3. Реализовать хранение обращений и решений в БД.
4. Подключить GPT для генерации текста решения (с жёстким парсингом результата: confirm / send_to_council / return_to_court).
5. Реализовать публикацию решения в канал и обновление статуса дела; уведомление суда/совета (по каналу или по тегу).

После этого можно повторять тот же подход для суда, совета, прокурора и КПП, с разными полномочиями и командами.

## 7. Резюме по боту «Старейшина»

| Элемент | Содержание |
|---------|------------|
| **Полномочия** | Только апелляции по процедуре, дела о референдуме, дела не установленные судом. |
| **Действия** | Подтвердить процесс; передать решение на исполнение в совет; один раз вернуть дело в суд с указанием нарушения; по референдуму — одобрить/отклонить или отправить на повтор при равных голосах. |
| **Ограничения** | Не отменять решения/прецеденты, не пересматривать существо права, не вмешиваться дважды в одно дело, не выносить приговоров. |
| **Команды** | `/appeal`, `/referendum_request`, `/elders_status`, `/elders_list` (ПМЖ/старейшина). |
| **Реализация** | Один оркестратор, общая БД дел, бот с проверкой ролей и типа дела, GPT для текста решения, публикация в Discord и передача дел в суд/совет. |

Если нужно, следующий шаг — детальная схема БД и список полей для дел/апелляций/референдумов под старейшин и суд.
